## Section D — Supervisor changes for multi-app shared pools

The current supervisor is single-app with hardcoded stages. Four surgical changes generalize it; nothing about the lease/reclaim/event-transaction core moves.

1. **Scheduler is stage-name-agnostic.** Today it branches on known stages. New: `SELECT … FOR UPDATE SKIP LOCKED` returns an `app_id`-tagged job; the supervisor looks up that app's manifest to resolve `stage → {run: code | agent, port?}`. Code stages dispatch to the plugin registry in-process; agent stages lease onto a matching pool worker.

2. **Pool manager keys pools by `(runtime, skill, model)`, not by app.** At boot it takes the *union* of all apps' `AgentSpec`s, dedups by pool key, and spawns `max(pool_size)` per key. A `feynman/entity-enrichment` pool serves the matchmaker and any future app that declares the same agent. Tool allowlists must be compatible per key (validated at load — mismatch = manifest error).

3. **DAG engine reads the saga list from the manifest.** When stage *k*'s event lands, enqueue stage *k+1* from that app's saga definition (instead of a hardcoded `next_stage` map). `on_reject` becomes a generic retry-edge in the DAG.

**Where hexagonal genuinely pays (≥2 real adapters or a real isolation seam) — already in the design:**
- `RuntimeLauncher` → `LocalProcessLauncher` (host) / `ContainerLauncher` (later). Two real adapters.
- App `ports:` (Matcher → LlmJudge / Embedding / GraphRag). The manifest's `ports:` section *literally is* the ports-and-adapters registry — hexagonal isn't new here, it's the name for what we already chose.
- Future `Sender`/`Scheduler` (dry_run vs real Gmail/Calendar, with compensations). Two adapters each.

**Where purist hexagonal would be a lie — the `Store`:** the saga model is *deliberately* Postgres-as-log. `SKIP LOCKED` leasing, `LISTEN/NOTIFY` wake-ups, and the single-transaction event+job+saga write are Postgres-specific and **load-bearing** — they're the whole reliability story. Hiding Postgres behind a pretend-swappable generic repository would add DTO/mapping ceremony for a second adapter that will never arrive, and would tempt someone to "simplify away" the exact SQL that makes crash-recovery correct. So I'd keep a `Store` **port** (so the domain core stays I/O-free and unit-testable against a fake), but let that port be *honestly Postgres-shaped* — it exposes `lease_ready_job()`, not `find_all()`.

**My recommendation: pragmatic hexagonal.** Domain core (saga engine, DAG advance, reconciler) imports zero I/O; everything it touches is a port; adapters plug in at the edge. Ports exist for **testability and real second-adapters**, not for imaginary DB portability.
